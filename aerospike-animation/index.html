How Aerospike Stores Data

An animated walk through the storage engine, then how AeroStream builds streams on top of it — digest → partition → index → rblocks → produce/consume/replay/pub-sub. Use the arrows or press →.

01 The data model hierarchy

Aerospike nests data in a fixed hierarchy. A namespace owns a storage engine and policy; a set groups records like a table; a record holds bins; each bin holds one typed particle. Records are schemaless — two records in a set can have totally different bins.

02 Key → digest → partition

A write supplies (namespace, set, key). Aerospike hashes set + key into a 20-byte cf_digest (RIPEMD-160). The low 12 bits of the first 4 bytes pick one of AS_PARTITIONS = 4096 partitions — the unit of distribution, migration, and reservation.

Why it matters: the partition is derived purely from the digest, so every node — and the client — can compute where a record lives with zero lookups. The client routes the request straight to that partition's master node.

03 The primary index (always in RAM)

Each partition owns an as_index_tree, split into lock-striped red-black sub-trees called sprigs. Each record is a 64-byte as_index node living in an arena (cf_arenax), referenced by a 40-bit handle, not a pointer. The index holds identity + version + a pointer to the data; the bins live in storage.

Hybrid memory model: the index (identity, TTL, version, data pointer) is always in RAM. The actual bin values can live in RAM, SSD, or pmem — addressed by file_id + rblock_id.

04 The storage layer — rblocks, wblocks, flat records

The index points at the data with file_id + rblock_id + n_rblocks. Storage is addressed in 16-byte rblocks and physically managed in large wblocks (write-block-size, e.g. 8 MB). A record is serialized to a packed as_flat_record (magic 0x037AF201) that carries its own digest and length.

Append-only: updating a record never overwrites bytes. It writes a new flat record elsewhere and re-points the index. The old copy becomes a dead rblock, reclaimed later by defrag.

One ack, two copies. In the default commit level the client's write isn't acknowledged until the replica(s) have it — durability and availability come from replication-factor, not a separate log.

06 Reclaim & expire — defrag and nsup

Because writes are append-only, wblocks fill with dead records. defrag copies the survivors out of a sparse wblock and frees it. Separately, the namespace supervisor nsup (nsup-period) sweeps the index, expiring records past their void_time and evicting oldest-TTL records when space runs low.

↓
survivors re-written, index re-pointed, wblock → FREE
nsup: TTL expiry by void_time
AeroStream uses this: every stream record is written with a TTL. nsup ages log records out with no compaction of its own — which is why the aerostream namespace must set nsup-period > 0.

07 The read path — one hop

A point read reserves the partition, walks the sprig's red-black tree by digest to the as_index, reads file_id + rblock_id, fetches the rblocks (from the post-write cache if hot, else the device), validates the as_flat_record magic, and unpacks the requested bins (as_flat_unpack_bins) back into typed particles.

Why p99 stays flat: a read is one in-RAM tree walk plus at most one device read. No B-tree descent on disk, no read amplification — that single-hop design is the whole latency story, even on SSD.

08 AeroStream — the broker is the database

AeroStream adds Kafka-style streams to this same engine. It claims 8 new as_proto message types (10–17) on the existing port 3000, dispatched from service.c into a self-contained module. A stream record is just an Aerospike record in the aerostream namespace — no separate broker, no extra hop.

The whole idea: stop running Kafka next to Aerospike. The log, the consumer offsets, and the records are all in the one process, on the one storage engine you already trust.

09 Produce — append to a partitioned log

STREAM_PRODUCE hashes the partition key (fnv32a % num_partitions) to pick a stream partition, assigns the next offset with a lock-free atomic (as_faa_uint64(&offset_seq)), then writes the record through the exact storage write path from stage 5 (an IOPS internal transaction). Offsets survive restart. The counter is in-memory (as_faa_uint64, one instruction, no I/O), and on restart it's rebuilt by an exponential + binary search over the existing log keys — so new writes never clobber old records.

10 Consume — a server-push session

STREAM_CONSUME opens a persistent session. A dedicated per-partition push thread holds the client fd open, reads records the same way as stage 7 (as_record_get → as_storage_rd_load_bins), and pushes STREAM_RECORD messages as they appear. The client sends STREAM_ACK to advance the committed offset and free an in-flight slot (max_in_flight).

Full duplex on one socket. The service thread reads your ACKs while the push thread writes records — the connection stays open for the life of the session, which is why this needs a dedicated thread, not the request/response pool.

11 Consumer groups & replay

Each group keeps its own committed offset per partition in the consumer_offsets set, so groups read the same log independently and at their own pace. STREAM_SEEK rewinds a group to any offset or timestamp. A timestamp seek is a binary search on the monotonic ts bin — O(log N) direct key reads, no secondary index.

Replay is not a copy. Seeking just moves a committed offset back and re-reads the records that are still in the log set. The data was never consumed-and-deleted — TTL is the only thing that removes it.

12 Pub/sub — ephemeral fan-out

STREAM_SUB registers a subscriber in an in-memory registry keyed by topic. On every produce, the record is fanned out live to all subscribers on that topic — using the same STREAM_RECORD format, but with no offsets and no durability. Disconnect and you miss whatever was sent meanwhile.
